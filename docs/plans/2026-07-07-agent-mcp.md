# Phase C — agent tools (MCP)

The tier-3 surface: a curated MCP toolset that lets a reasoning agent **see** the market and
**research** strategies, while the deterministic gate still decides. Agents propose and explain;
they never touch the money path directly.

## Result

`trading-mcp` runs in compose over SSE. An MCP client (`scripts/mcp_verify.py`) drives the agent's
honest research loop end-to-end:

```
tools: list_strategies, account, positions, market_snapshot, bars, backtest,
       evaluate_gate, recent_decisions          (no place/order tool)
market_snapshot BTC/USD → latest price 63614.1
backtest sma_crossover  → return -5.1%, sharpe -0.27
evaluate_gate sma_crossover → passed=False (OOS sharpe -1.01, -11.28% return, 25% folds)
```

An agent can **see** the market and **research** strategies, ending in a gate verdict — and has no
tool to place an order. The propose-not-execute boundary is enforced by the tool surface itself
(a test asserts no `place`/`order` tool is exposed).

**Deltas from plan:** the MCP is a second entrypoint in the orchestrator image (compose runs it
with a different command), reusing the whole core; added public `get_positions_raw` /
`get_snapshot_raw` gateway reads rather than reaching through private methods.

## Design decision — Python MCP in the orchestrator

`h`'s MCP servers are TS; ours is Python, inside the orchestrator app. Reason: the valuable agent
tools here are the **research** ones — backtest, walk-forward gate, decisions history — which are
Python. A TS MCP could only proxy gateway reads; a Python MCP reuses the whole intelligence core
and keeps that layer one language. It runs as a second presentation entrypoint over the existing
`application`/`infrastructure`, using `mcp`'s `FastMCP` (SSE transport).

## The curated toolset (read + research; NO placement)

Per the architecture ("agents get read-heavy tools; never raw order placement"):

- **Market visibility** (proxy the gateway): `market_snapshot(symbol)`, `bars(symbol, timeframe,
  limit)`, `account()`, `positions()`, `clock()`.
- **Research** (reuse the intelligence core): `list_strategies()`,
  `backtest(strategy, symbol, timeframe, bars)` → metrics, `evaluate_gate(strategy, symbol)` →
  the walk-forward verdict, `recent_decisions(limit)` → the journal.

**No `place_order` tool, by design.** An agent's influence path is: research a strategy → if it
clears the gate, propose it for a `strategy_tick` run. The workflow re-runs the binding gate before
placing, so an agent can never route around the deterministic money path. Triggering a
`strategy_tick` (which is itself gated) can be a later, deliberately-added tool; this phase is
read + research only.

## Milestone 1 — tool functions (testable)

`presentation/mcp_tools.py`: async functions returning plain JSON-serializable dicts (Result mapped
to `{ "error": ... }` at this edge — agents get data, not exceptions), reusing gateway/backtest/
walkforward/gate/journal. Unit-test the pure-shaped ones (list_strategies, error mapping); the
gateway-backed ones are covered by the live check.

## Milestone 2 — server + live verify

`presentation/mcp_server.py` registers the tools with `FastMCP` and serves SSE. Run as a
`trading-mcp` compose service (same image, different command). Verify by driving the tools live
(`call_tool` against the running gateway) — snapshot → backtest → evaluate_gate — the agent's
honest research loop, ending in a gate verdict, no order.

## Non-goals

No LLM agent wired yet (that consumes this surface; needs keys + is nondeterministic — a follow-on),
no order-placement or strategy-trigger tool this phase, no prompt/role design.
